# Todo List 工具详细开发计划

## 1. 项目目标

基于对 `Todoist`、`Microsoft To Do`、`Things 3` 等产品的对比，这个产品的第一阶段应明确定位为：

`一个面向个人使用的、轻量级任务记录工具，支持任务拆解、分类管理和快速录入。`

这意味着产品目标不是做“大而全”的效率平台，而是优先解决下面 3 个核心问题：

- 用户能否快速记下一个任务
- 用户能否把任务继续拆成更小的执行单元
- 用户能否用简单直观的分类和层级管理这些任务

## 2. 产品原则

开发过程中建议始终遵守以下原则：

- 轻量优先：优先保证记录、查看、完成任务这条主链路足够顺畅
- 低学习成本：避免让用户先理解系统，再开始记任务
- 层级清晰：支持“分类 -> 任务 -> 子任务”的自然结构
- 渐进增强：先做核心体验，再逐步补充提醒、排序、截止日期等增强功能
- 克制功能范围：第一版不做团队协作、复杂自定义、多视图系统

## 3. 推荐产品结构

结合报告结论，建议采用下面这套信息结构：

`分类 -> 任务 -> 子任务`

说明：

- 分类：用于表达生活领域，如工作、学习、生活、购物
- 任务：用户要完成的具体事项
- 子任务：用于拆解任务的执行步骤或更小动作

这样做的好处：

- 比“任务 -> 步骤”更有层级感
- 比纯树状无限层级更简单，认知负担更低
- 更适合个人待办场景，能兼顾易用性和拆解能力

## 4. 用户画像与核心场景

### 目标用户

- 以个人用户为主
- 需要记录生活、学习、工作中的待办事项
- 希望任务可以被拆小，但不希望使用复杂项目管理软件

### 核心使用场景

1. 快速添加一个任务
2. 为任务选择分类
3. 将任务拆成 2 到 5 个子任务
4. 在“今日”视图中处理当天事项
5. 完成任务后及时勾选，获得清晰反馈
6. 通过搜索快速找到历史任务

## 5. MVP 功能范围

### 必须做

- 分类管理
  - 创建分类
  - 编辑分类名称
  - 删除分类
- 任务管理
  - 新建任务
  - 编辑任务标题和备注
  - 标记完成 / 未完成
  - 删除任务
- 子任务管理
  - 为任务添加子任务
  - 标记子任务完成 / 未完成
  - 展开 / 收起子任务
- 基础视图
  - 今日
  - 全部
  - 已完成
  - 按分类查看
- 搜索
  - 支持按任务标题搜索
- 基础排序
  - 按创建时间排序
  - 手动调整同级任务顺序

### 应该做

- 截止日期
- 重要标记 / 置顶
- 快速录入入口
- 空状态和引导文案
- 本地持久化或账号数据存储

### 不进入第一版

- 团队协作
- 评论、附件
- 番茄钟、习惯打卡
- 复杂标签系统
- 看板 / 日历 / 甘特图等多视图
- 高自由度自定义字段
- 自动化规则系统

## 6. 核心页面规划

### 6.1 首页 / 主工作台

目标：让用户一进入应用就能看到自己的待办主界面。

建议布局：

- 左侧：分类列表、基础视图入口
- 中间：任务列表
- 右侧或弹层：任务详情编辑区

核心模块：

- 顶部快速新增输入框
- 分类切换
- 任务列表
- 子任务展开区
- 搜索框

### 6.2 今日视图

目标：聚焦当天要做的任务。

展示规则建议：

- 显示截止日期为今天的任务
- 显示被用户手动标记为“今天处理”的任务
- 显示未完成任务

### 6.3 全部任务视图

目标：用于总览和回顾所有待办。

建议支持：

- 按分类分组
- 按完成状态筛选
- 按截止日期排序

### 6.4 任务详情面板

用于承载单个任务的完整信息。

字段建议：

- 标题
- 分类
- 备注
- 截止日期
- 是否重要
- 子任务列表
- 创建时间
- 完成状态

## 7. 关键交互设计

### 快速录入

设计目标：

- 用户在 3 秒内可以新增一条任务
- 默认只输入标题即可创建
- 更多字段放入“展开后编辑”

### 任务拆解

设计目标：

- 用户创建任务后可以立即添加子任务
- 子任务录入方式应足够轻，不要跳转太多层
- 展开和收起状态要明显

### 完成反馈

设计目标：

- 勾选完成后立即有视觉反馈
- 已完成任务默认弱化展示
- 支持恢复未完成

### 分类管理

设计目标：

- 分类数量应少而清晰
- 默认提供几个示例分类，降低首次使用门槛

## 8. 数据模型设计建议

如果你做 Web 或桌面应用，第一版可以先采用如下核心实体：

### Category

- `id`
- `name`
- `color`
- `sort_order`
- `created_at`
- `updated_at`

### Task

- `id`
- `title`
- `description`
- `category_id`
- `status`
- `priority`
- `due_date`
- `sort_order`
- `is_today`
- `is_starred`
- `created_at`
- `updated_at`
- `completed_at`

### Subtask

- `id`
- `task_id`
- `title`
- `status`
- `sort_order`
- `created_at`
- `updated_at`
- `completed_at`

### 可选说明

- 第一版建议只支持 1 层子任务，不做无限嵌套
- 这样能明显降低数据复杂度、交互复杂度和后续 bug 风险

## 9. 状态流转规则

建议先把规则定清楚，避免开发后期反复修改。

### 任务状态

- `todo`
- `completed`

### 子任务状态

- `todo`
- `completed`

### 规则建议

- 任务完成后，子任务默认同时视为完成，或提示用户是否同步完成
- 如果一个任务下存在未完成子任务，任务仍允许被用户手动标记完成
- 已完成任务支持恢复，恢复后子任务状态保持原样或一并恢复，需在产品中固定规则

第一版更推荐的规则是：

- 任务和子任务状态允许独立
- 用户手动标记任务完成时，不强制改写全部子任务
- 界面提示“该任务仍有未完成子任务”

这样更灵活，也更接近真实使用场景。

## 10. 技术方案建议

如果你还没有确定技术栈，建议第一版优先选择开发效率高、便于快速验证的方案。

### 方案 A：Web 应用优先

适合：

- 想先快速做 MVP
- 想覆盖桌面和移动端浏览器
- 想后续再封装桌面端

推荐组合：

- 前端：`Next.js` 或 `React + Vite`
- UI：简洁自定义组件，避免一开始重度依赖复杂后台模板
- 状态管理：`Zustand` 或 React 原生状态
- 数据存储：
  - 单机 MVP：`SQLite` / `IndexedDB`
  - 有登录需求：`PostgreSQL`
- ORM：`Prisma`

### 方案 B：桌面优先

适合：

- 更偏个人工具
- 更关注桌面端使用体验

推荐组合：

- 前端：`React`
- 桌面容器：`Tauri` 或 `Electron`
- 本地数据库：`SQLite`

### 当前更推荐

优先做 `Web MVP`，原因：

- 开发成本更低
- 迭代更快
- 更容易先验证产品结构是否成立

## 11. 开发阶段拆分

### 第一阶段：产品定义与原型

目标：把方向定死，避免边做边改。

输出物：

- 产品 PRD
- 页面信息架构图
- 核心页面低保真原型
- 数据结构草案
- 交互规则清单

重点确认：

- 是否只支持一层子任务
- 今日视图的规则
- 任务完成和子任务完成的状态规则
- 分类是否允许自定义颜色

### 第二阶段：MVP 开发

目标：完成最小可用版本。

开发任务：

1. 搭建项目基础结构
2. 完成分类 CRUD
3. 完成任务 CRUD
4. 完成子任务 CRUD
5. 完成今日 / 全部 / 已完成视图
6. 完成搜索
7. 完成基础排序
8. 完成持久化存储

验收标准：

- 用户可完整创建分类、任务、子任务
- 数据刷新后不丢失
- 能顺畅完成“记录 -> 拆解 -> 完成”闭环

### 第三阶段：可用性优化

目标：让 MVP 从“能用”提升到“好用”。

优化内容：

- 快速录入交互优化
- 键盘快捷操作
- 拖拽排序
- 空状态优化
- 动画和反馈优化
- 移动端适配

### 第四阶段：增强版本

目标：在不破坏轻量定位的前提下做增强。

可选能力：

- 提醒
- 重复任务
- 归档
- 数据导出
- 登录与多端同步

## 12. 推荐开发优先级

建议按照下面顺序开发：

1. 数据模型
2. 分类列表
3. 任务列表
4. 新建任务
5. 任务详情编辑
6. 子任务能力
7. 完成状态切换
8. 今日 / 已完成视图
9. 搜索
10. 排序与体验优化

原因：

- 先做数据骨架，后面改动成本最低
- 先把主流程跑通，再补辅助能力
- 搜索和排序虽然重要，但不应阻塞核心记录流程

## 13. 里程碑建议

### 里程碑 1：结构验证版

周期建议：1 周

目标：

- 验证“分类 -> 任务 -> 子任务”结构是否顺手
- 用假数据或本地数据跑通主流程

### 里程碑 2：MVP 可用版

周期建议：2 到 3 周

目标：

- 完成核心功能闭环
- 可以交给少量真实用户试用

### 里程碑 3：体验优化版

周期建议：1 到 2 周

目标：

- 优化录入效率和信息组织
- 提升视觉层次和细节反馈

## 14. 风险与规避建议

### 风险 1：功能膨胀

表现：

- 很快加入标签、日历、番茄钟、协作、附件

规避：

- 每加一个功能都问一句：它是否直接服务“快速记录 + 任务拆解”

### 风险 2：层级过深

表现：

- 任务支持无限嵌套，导致操作复杂、渲染复杂、状态复杂

规避：

- 第一版只支持一层子任务

### 风险 3：信息结构不稳定

表现：

- 一会儿按分类组织，一会儿按项目组织，后期频繁返工

规避：

- 第一阶段先固定“分类 -> 任务 -> 子任务”模型

### 风险 4：交互过重

表现：

- 创建一条任务要填很多字段

规避：

- 默认只填标题，其他字段延后填写

## 15. 成功指标建议

第一版可以关注这些指标：

- 用户创建首条任务所需时间
- 平均每个任务的子任务数量
- 今日视图的使用频率
- 完成任务比例
- 7 天留存

如果是自己先做验证，也可以先关注定性反馈：

- 用户是否觉得“记任务很快”
- 用户是否觉得“拆任务很自然”
- 用户是否觉得“分类清晰，不乱”

## 16. 最终建议

这个产品最合适的切入方式不是做复杂任务管理，而是做一个：

`足够轻、足够快、能自然拆解任务的个人 Todo 工具。`

第一版最重要的不是功能多，而是把下面这条链路做顺：

`快速记录 -> 分类归档 -> 拆成子任务 -> 在今日视图执行 -> 完成勾选`

只要这条主链路体验足够顺畅，你的产品就已经具备明确价值。后续增强功能都应建立在这条链路稳定成立的基础上。
