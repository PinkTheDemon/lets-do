# Todo List 工程化开发计划

## 1. 开发目标

我会先把这个项目当成一个需要快速验证价值的个人效率产品，而不是一个长期堆功能的平台。

第一阶段只解决一件事：

`让用户可以快速记录任务，并把任务自然拆成子任务进行管理。`

因此开发目标不是“做全”，而是尽快做出一个稳定、顺手、可验证的 MVP。

## 2. 先定义边界

### 做什么

- 面向个人用户
- 支持分类
- 支持任务
- 支持一层子任务
- 支持今日 / 全部 / 已完成视图
- 支持快速录入和基础搜索

### 不做什么

- 不做协作
- 不做复杂项目管理
- 不做无限层级任务树
- 不做复杂标签系统
- 不做日历、番茄钟、习惯打卡
- 不做高自由度自定义字段

原因很简单：

- 这些功能都会显著拉高产品复杂度
- 它们不直接服务当前核心价值
- 它们会拖慢 MVP 上线速度

## 3. 我会如何拆开发计划

如果是我自己来开发，我会按 5 个层次组织：

1. 产品结构定型
2. 数据模型落地
3. 主流程开发
4. 体验优化
5. 增强能力评估

这比按页面或按功能列表平铺更好，因为它更符合真实开发顺序，也更方便控制返工。

## 4. 阶段一：产品结构定型

这一阶段不急着写太多代码，核心是把最容易反复修改的东西定下来。

### 4.1 固定信息架构

我会直接固定为：

`分类 -> 任务 -> 子任务`

不在这个阶段引入“项目”“标签树”“多层嵌套任务”这些额外结构。

### 4.2 固定主流程

主流程只有一条：

`新增任务 -> 选择分类 -> 补充子任务 -> 在列表中查看 -> 标记完成`

只要这条链路顺，产品第一版就成立。

### 4.3 固定关键规则

我会优先拍板这些规则：

- 子任务只支持 1 层
- 创建任务时只要求输入标题
- 任务和子任务状态允许独立
- 今日视图先按 `due_date = today` 和 `is_today = true` 两个条件实现
- 删除默认走软删除或二次确认，避免误操作

### 4.4 阶段产出

- 一份简化 PRD
- 页面草图
- 实体关系设计
- 关键交互规则表

## 5. 阶段二：数据模型落地

这个阶段我会先把“以后最难改”的部分做好，也就是数据结构和状态规则。

### 5.1 核心实体

#### Category

- `id`
- `name`
- `color`
- `sort_order`
- `created_at`
- `updated_at`

#### Task

- `id`
- `title`
- `description`
- `category_id`
- `status`
- `due_date`
- `is_today`
- `is_starred`
- `sort_order`
- `created_at`
- `updated_at`
- `completed_at`

#### Subtask

- `id`
- `task_id`
- `title`
- `status`
- `sort_order`
- `created_at`
- `updated_at`
- `completed_at`

### 5.2 状态设计

我会避免过早设计很多状态。

任务和子任务第一版只保留：

- `todo`
- `completed`

这样做的好处：

- 查询简单
- 前端渲染简单
- 状态切换逻辑简单
- 后续需要扩展时也不难兼容

### 5.3 存储策略

如果目标是快速开发，我会这样选：

- Web MVP：`React + SQLite/PostgreSQL`
- 单机优先：本地 `SQLite`
- 如果先不上账号：本地持久化即可

我的原则是：

- 先保证数据可靠
- 再考虑同步
- 最后再考虑复杂跨端

## 6. 阶段三：主流程开发

这是最关键的一段，我会严格按用户使用频率从高到低开发。

### 6.1 第一个迭代：可录入

先完成：

- 分类列表
- 新建任务
- 任务列表展示
- 标记完成

这个迭代的目标不是完整，而是验证“用户能不能开始用”。

验收标准：

- 用户可以在 10 秒内创建第一条任务
- 创建后的任务立刻出现在列表中
- 刷新后数据不丢失

### 6.2 第二个迭代：可拆解

再补上：

- 编辑任务
- 添加子任务
- 子任务完成切换
- 展开 / 收起子任务

这个迭代完成后，产品核心价值才真正成立。

验收标准：

- 用户能把一个任务拆成多个子任务
- 列表结构清晰，不混乱
- 任务和子任务状态展示一致

### 6.3 第三个迭代：可组织

加入：

- 今日视图
- 全部视图
- 已完成视图
- 按分类筛选
- 标题搜索

这个阶段解决“记录完之后怎么管理”。

验收标准：

- 用户能快速找到任务
- 用户能区分今天要做什么
- 已完成任务不影响主列表阅读

### 6.4 第四个迭代：可顺手使用

加入：

- 快速录入优化
- 回车新增
- 键盘焦点切换
- 同级排序
- 重要标记
- 截止日期

这个阶段不是补业务闭环，而是补“效率感”。

## 7. 阶段四：体验优化

如果 MVP 已经能用，我下一步不会立刻加大功能，而是先做体验打磨。

### 7.1 优先优化的点

- 创建任务步骤是否太长
- 子任务展开是否清晰
- 完成态是否足够明确
- 空列表时是否有引导
- 移动端是否能正常操作

### 7.2 我会重点关注的体验指标

- 首条任务创建耗时
- 平均每次新增任务点击次数
- 每个任务的平均子任务数
- 今日视图使用率
- 完成率

如果这些数据不好，再多新功能也没有意义。

## 8. 阶段五：增强能力评估

只有在 MVP 已经稳定后，我才会评估下一批功能。

### 可以进入候选池的功能

- 提醒
- 重复任务
- 归档
- 导出
- 多端同步

### 暂时继续压住的功能

- 团队协作
- 评论
- 附件
- 标签体系扩展
- 多视图系统

我的判断标准是：

这个功能是否显著提升“快速记录 + 自然拆解”的效率。

如果不是，就不该优先做。

## 9. 我会怎么安排开发节奏

如果是一个人开发，我会按下面节奏推进：

### 第 1 周

- 定义产品结构
- 画低保真页面
- 确定技术栈
- 搭项目骨架
- 建数据库模型

### 第 2 周

- 做分类模块
- 做任务 CRUD
- 做任务列表
- 做完成态切换

### 第 3 周

- 做子任务模块
- 做今日 / 已完成视图
- 做搜索
- 做基础样式

### 第 4 周

- 做排序和快速录入
- 修交互细节
- 补测试
- 修 bug
- 输出 MVP

## 10. 我会怎么定义验收标准

开发计划不能只有任务，没有验收标准，否则会一直“看起来快做完了”。

我会给 MVP 定 5 条硬标准：

1. 用户能在 3 秒内打开界面并开始录入
2. 用户能在 10 秒内创建“分类 + 任务”
3. 用户能在 20 秒内把任务拆成 2 到 3 个子任务
4. 用户能在今日视图找到今天要处理的任务
5. 刷新、重开应用后数据不丢失

只要这 5 条没过，就不进入功能扩展阶段。

## 11. 技术执行建议

如果是我直接开工，我会优先选这套：

- 前端：`React + Vite`
- 样式：轻量自定义 CSS 或 Tailwind
- 状态管理：`Zustand`
- 数据层：`SQLite + Prisma`

原因：

- 足够快
- 足够稳
- 适合一个人快速迭代
- 不会过早把精力耗在基础设施上

如果你更偏桌面端，我会把外层包成 `Tauri`，但不会一开始就上很重的桌面框架能力。

## 12. 最后结论

如果让我来组织这个项目，我不会先写一份很长的“功能大全”，而会先把开发计划压缩成一句话：

`先做一个最顺手的任务记录和拆解工具，再决定是否扩张。`

对应到执行上，就是：

- 先定结构，不摇摆
- 先做数据模型，不返工
- 先打通主流程，不堆功能
- 先优化体验，不急扩张
- 先验证价值，再谈平台化

这套组织方式的核心不是“全面”，而是“让你尽快得到一个可用、可验证、可继续演进的版本”。
